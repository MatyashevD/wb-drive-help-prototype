# WB Drive · Раздел «Помощь» — кликабельный прототип

Прототип помощи в мобильном приложении WB Drive. Сделан в чистом HTML/CSS/JS — без сборки, без зависимостей. Для тестирования с сотрудниками.

## Что внутри

Прототип показывает архитектурный подход **«нативный экран + WebView»**:

- **Профиль** (entry) → пункт «Помощь»
- **Помощь** (нативный экран):
  - 🔍 поиск по статьям (клиентский, не зависит от CMS)
  - 🎯 featured-карточка «Начало работы» (визуальный гайд для новичков)
  - 📂 6 категорий: Маршруты, Оплата, Документы, Возвраты, Аккаунт, Техподдержка
  - 🔥 топ-5 популярных вопросов
  - 💬 CTA «Не нашли ответ?» → чат с поддержкой
- **Категория** (нативный экран): список статей категории
- **Статья** (имитация WebView): контент из CMS, в правом верхнем углу бейдж `webview · cms`, чтобы продакт/QA видели, что это WebView

Все статьи — мок из `data.js`. В проде они пришли бы по `GET /help/articles` (метаданные для поиска и навигации) + `GET /help/articles/:id` (HTML для WebView).

## Файлы

| Файл | Назначение |
| --- | --- |
| `index.html` | Точка входа, мобильный фрейм, SVG-спрайт иконок |
| `styles.css` | Дизайн-токены 1:1 с Figma DS UI-kit (цвета, радиусы, типография) |
| `app.js` | Hash-роутер, рендер всех экранов, поиск с подсветкой совпадений |
| `data.js` | Мок CMS-контента (6 категорий + 19 статей) |

## Локальный запуск

```bash
# Любой статический сервер. Например:
python3 -m http.server 8000
# или
npx serve .
```

Затем откройте `http://localhost:8000` в Chrome → DevTools → Toggle Device Toolbar (Cmd+Shift+M) → выберите iPhone/Pixel.

## Деплой на GitHub Pages

У вас уже есть аккаунт и репозиторий — вот короткая инструкция.

### Если хотите выложить в существующий репо

1. Скопируйте все файлы (`index.html`, `styles.css`, `app.js`, `data.js`) в нужную папку репозитория.
   - Удобно: создайте подпапку, например `prototypes/help/`.
2. Запушьте изменения в `main`.
3. На GitHub: **Settings → Pages**.
4. **Source**: `Deploy from a branch`.
5. **Branch**: `main`, **Folder**: `/` (если файлы в корне) или нужная подпапка не подойдёт через UI — для подпапки используйте трюк с веткой `gh-pages` (см. ниже).
6. Через 1–2 минуты ссылка вида `https://<username>.github.io/<repo>/` начнёт открываться.

### Вариант с отдельной веткой `gh-pages` (если файлы лежат в подпапке)

```bash
# Допустим, файлы в prototypes/help/
git subtree push --prefix prototypes/help origin gh-pages
```

После этого в **Settings → Pages** выберите **Branch: gh-pages, Folder: /**.

### Вариант с отдельным репо (рекомендую — чище и проще)

```bash
# Создайте пустой репо на GitHub: wb-drive-help-prototype (Public)
git init
git add .
git commit -m "Help prototype: native screen + WebView mock"
git branch -M main
git remote add origin https://github.com/<username>/wb-drive-help-prototype.git
git push -u origin main
```

Затем **Settings → Pages → Source: Deploy from a branch → Branch: main, Folder: /** → готово.

Финальная ссылка: `https://<username>.github.io/wb-drive-help-prototype/`

### Что показать сотрудникам

Откройте ссылку с телефона. Если делитесь в чате — на десктопе фрейм будет красивый (телефон по центру), на мобиле — на весь экран.

## Что дальше для прода

1. **CMS API.** Добавить два эндпоинта:
   - `GET /help/articles` — массив объектов `{ id, categoryId, title, excerpt, tags, popularity, updatedAt }`. Этот список приложение кэширует на 24 часа и использует для поиска и навигации.
   - `GET /help/articles/:id` — HTML контент. Загружается в WebView только при открытии статьи.
2. **Аналитика.** Логировать `help_search_query`, `help_article_open` (с `articleId`), `help_support_clicked`. По этим событиям формировать `popularity` обратно в CMS.
3. **Локализация.** Если поедем в СНГ — добавить `lang` в API + переключатель в шапке.
4. **Офлайн.** Кэшировать список статей в `localStorage`, чтобы поиск работал без сети (особенно полезно водителям между точками).

## Архитектурное решение, зафиксированное в прототипе

> **Гибрид B + C из обсуждения**: нативный экран помощи (поиск, категории, популярные вопросы, featured) + WebView для контента статьи.
>
> Поиск делаем клиентский — это снимает требование «CMS должна уметь в поиск». Приложение раз в N часов забирает метаданные статей и ищет по ним локально. WebView грузит только конкретную статью, что быстро и удобно для контент-команды (правки без релиза).
