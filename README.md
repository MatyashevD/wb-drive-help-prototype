# WB Drive · Прототипы UX

Кликабельные HTML-прототипы для мобильного приложения WB Drive. Без сборки, без зависимостей — чистый HTML/CSS/JS поверх единой дизайн-системы.

## Что внутри

| Прототип | URL | Кейс |
|---|---|---|
| **Помощь — нативный + WebView (Вариант B, согласовано в реализацию)** | `/` (`index.html`) | Нативный экран с табами и поиском, тело статьи открывается webview-ссылкой от CE RWB |
| **Помощь — одна WebView-простыня (Вариант A, архивный)** | `/webview.html` | Альтернатива «вся помощь = одна страница из CMS». Оставлено как контекст принятого решения |
| **Кнопка «Завершить поездку» (Склад→ПВЗ)** | `/finish-route.html` | Защита от случайных нажатий + поддержка ЧП-кейсов (поломка, ДТП) + flow возвратов на склад. Сравнение трёх паттернов (A/B/C) |

# Раздел 1. Прототипы «Помощи»

Два альтернативных подхода к разделу Помощи в мобильном приложении WB Drive. Один источник контента (`data.js`) — два разных UI поверх него, можно переключаться между ними и показывать сотрудникам обе версии.

## Прототипы

| | Прототип A · WebView | Прототип B · Нативный + WebView |
|---|---|---|
| **URL** | `/webview.html` | `/` (главная) |
| **Идея** | Из CMS приходит одна ссылка, страница — длинная простыня с аккордеонами, картинками, перекрёстными ссылками | Нативный экран с табами и поиском, WebView только для контента статьи |
| **Разработка** | Минимум — на стороне приложения только «открой WebView с этой ссылкой» | Больше нативного кода: tab-router, поиск, аналитика |
| **Поиск** | Полнотекстовый по странице (наша добавка поверх CMS) | Клиентский по метаданным статей |
| **Контент-операции** | Полная свобода CMS-редактора (аккордеоны, картинки, ссылки) | Контент в CMS, навигация в коде |
| **UX скорость** | Долгий скролл; поиск спасает | 1 тап до ответа на «Популярном» табе |
| **Аналитика** | По странице целиком | По каждой статье и категории отдельно |

### Прототип A — «Помощь = одна ссылка из CMS»

Минимум разработки. Приложение получает один URL и открывает его в WebView. Внутри:
- Длинная страница с заголовками разделов (Маршруты / Оплата / Документы / Возвраты / Аккаунт / Техподдержка)
- В каждом разделе — статьи в виде нативных HTML-аккордеонов (`<details>`)
- Внутри статей: текст, иллюстрации, перекрёстные ссылки на другие статьи (якорная навигация)
- Поверх — поле «Найти на странице»: фильтрует аккордеоны и подсвечивает совпадения

Все возможности CMS-редактора используются на 100%: добавил картинку — она появилась, добавил ссылку на другую статью — работает.

### Прототип B — «Нативный экран + WebView для статьи» (рекомендую)

- **Профиль** (entry) → пункт «Помощь»
- **Помощь** — нативный экран:
  - 🔍 поиск (клиентский по метаданным статей)
  - 📑 sticky-табы: «Популярное · Маршруты · Оплата · Документы · Возвраты · Аккаунт · Техподдержка»
  - На дефолтном табе «Популярное» — featured «Начало работы» + топ-7 вопросов по `popularity`
  - На категориях — список статей раздела
  - 💬 CTA «Не нашли ответ?» → чат с поддержкой
- **Статья** — имитация WebView с бейджем `webview · cms` в углу. В проде CMS отдаёт HTML по `GET /help/articles/:id`.

## Архитектура для прода (общее для обоих)

| Эндпоинт | Возвращает | Используется |
|---|---|---|
| `GET /help/articles` | `[{id, categoryId, title, excerpt, tags, popularity, updatedAt}]` | Только для прототипа B (поиск, навигация) |
| `GET /help/articles/:id` | HTML тела статьи | Оба прототипа (контент в WebView) |
| `GET /help/page` | HTML простыни целиком | Только прототип A |

## Файлы

| Файл | Назначение |
| --- | --- |
| `index.html` | **Прототип B** — нативный экран Помощи (точка входа: профиль → Помощь) |
| `app.js` | Hash-роутер, табы, поиск, рендер экранов прототипа B |
| `webview.html` | **Прототип A** — длинная WebView-простыня с аккордеонами |
| `webview.js` | Рендер простыни, полнотекстовый поиск по странице, якорная навигация |
| `styles.css` | Общие дизайн-токены и компоненты (цвета, радиусы, типография, list-item, navbar, и т.д.) |
| `data.js` | Единый мок CMS-контента (6 категорий + 19 статей). **Используется обоими прототипами**, чтобы они всегда показывали одни и те же данные |

## Что дальше для прода

1. **CMS API.** Добавить два эндпоинта:
   - `GET /help/articles` — массив объектов `{ id, categoryId, title, excerpt, tags, popularity, updatedAt }`. Этот список приложение кэширует на 24 часа и использует для поиска и навигации.
   - `GET /help/articles/:id` — HTML контент. Загружается в WebView только при открытии статьи.

## Архитектурное решение, зафиксированное в прототипе

> **Гибрид B + C из обсуждения**: нативный экран помощи (поиск, категории, популярные вопросы, featured) + WebView для контента статьи.
>
> Поиск делаем клиентский — это снимает требование «CMS должна уметь в поиск». Приложение раз в N часов забирает метаданные статей и ищет по ним локально. WebView грузит только конкретную статью, что быстро и удобно для контент-команды (правки без релиза).

# Раздел 2. Прототип кнопки «Завершить поездку»

**URL:** `/finish-route.html`

## Кейс

На поездке Склад → ПВЗ (8–15 точек разгрузки) водители-новички иногда **по незнанию тапают большую primary-кнопку «Завершить поездку»** на табе «Возвраты», не успев разгрузить все коробки. Это закрывает рейс с потерями, и водитель остаётся без понимания того, что произошло.

Параллельные кейсы, которые **нельзя** ломать защитой:

- **ЧП:** водитель должен иметь возможность завершить поездку досрочно (поломка, ДТП, проблема со здоровьем).
- **Возвраты:** если водитель забрал возвратные коробки с ПВЗ, поездку нельзя закрывать до сдачи их обратно на склад.

## Решение в прототипе

Три уровня защиты, ни один из которых не блокирует ЧП:

1. **Адаптивная нижняя плашка.** Primary-кнопка «Завершить поездку» появляется только когда поездка реально готова к закрытию (все ПВЗ выгружены и, если есть возвраты, они сданы на складе). До этого момента — primary занят «Навигатором», а доступ к досрочному завершению — отдельной маленькой кнопкой.
2. **Прогресс-бар** `0 / 44` в карточке. Заполняется по мере разгрузки, в конце становится зелёным.
3. **Bottom sheet «Что произошло?»** с 3 опциями:
   - **Случилось ЧП** → уведомить логиста (событие в систему + чат с логистом в WhatsApp/Telegram) + опциональный звонок в 112 + slide-to-confirm для закрытия поездки.
   - **Не могу продолжать поездку** → список неразгруженных точек, перечисление возвратных коробок (если есть), warning «эти коробки остаются за вами, довезите их на склад до конца смены» + slide-to-confirm.
   - **Я нажал случайно** → возврат к поездке без действия (honest-exit для новичков).

Когда поездка реально завершена — bottom sheet с причинами не открывается, primary-кнопка отрабатывает по одному тапу.

## Flow возвратов

- На табе «Возвраты» — карточка склада с адресом и `0/55`. Кнопка «Разгрузить» (в реале — QR + сканер коробок, в прототипе упрощено) → водитель помечен как «на складе» → primary-кнопка переключается в режим «Сдать возвраты и завершить».
- Если водитель пытается завершить поездку с невыгруженными возвратами не на складе — отдельный warning-sheet «Везите возвраты на склад» с CTA «Построить маршрут до склада».

## Сравнение вариантов в прототипе

В демо-панели снизу — переключатель «Завершить досрочно»: `A · ссылка` / `B · иконка ⚠` / `C · «!» + ревью`.

| Вариант | Где живёт «Завершить досрочно» | Bottom sheet | Заметность для новичка |
|---|---|---|---|
| **A · ссылка** | Мелкая текстовая ссылка `⚠ Завершить досрочно ›` над «Навигатором» | «Что произошло?» — 3 причины (ЧП / Не могу продолжать / Случайно) | Средняя |
| **B · иконка ⚠** | Иконка-кнопка `⚠` (мягкий красный фон) слева от «Навигатора» | То же, что у A | Низкая (легко пропустить) |
| **C · «!» + ревью** | Постоянная кнопка `!` (белая, красная обводка) слева от «Навигатора» | Единый упрощённый шит «У вас что-то случилось?»: инфо + «Написать в поддержку» + slide-to-confirm | Средняя |

Вариант C — результат дизайн-ревью под ограничения бека (одно событие вместо 3 веток). A и B оставлены для сравнения с дизайнером.

## Демо-режим

Внизу страницы — переключатель состояний:
- **Прогресс поездки:** `0/44 → 12/44 → 42/44 → 44/44`.
- **Возвраты:** «есть / нет» (управляет наличием возвратных коробок).
- **На складе:** «да / нет» (имитирует приезд на склад и сдачу возвратов).
- **Завершить досрочно:** `A · ссылка` / `B · иконка ⚠` / `C · «!» + ревью` (переключает паттерн).

## Что в проде

- Состояние кнопки вычисляется из прогресса: `pvz_done === pvz_total && (returns_total === 0 || returns_delivered === returns_total)` → primary, иначе — secondary/ссылка.
- Slide-to-confirm — нативный компонент (`SwipeButton` в Material, кастом на iOS).
- ЧП-причина → событие `route_emergency_initiated` (пуш логисту через бэкенд + строка в чате) + опциональный `tel:112`.
- Плановое досрочное завершение → событие `route_planned_finish` с причиной, количеством неразгруженных коробок и наличием возвратов.
- Завершение из стейта «все ПВЗ выгружены, возвраты не сданы вне склада» → warning-sheet с CTA «Построить маршрут до склада».
